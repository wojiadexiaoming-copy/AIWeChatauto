# 微信公众号草稿上传与标题截断修复方案

> 📌 **本案旨在记录并在项目开发中重点标记微信公众号草稿接口在 Windows 环境下出现 `40007` (封面 ID 无效) 及 `45003` (标题长度超限) 的问题机理与彻底解决方案，防范后续开发重构中再次引入同类 Bug。**

---

## 1. 问题现场与现象

### 现象 A：`errcode: 40007, errmsg: invalid media_id`
*   **现象**：将文章保存为草稿时，微信接口返回封面图 ID (`thumb_media_id`) 无效。然而，若使用测试工具单独查询该封面 `media_id`，发现其在微信后台完全合法且有效。
*   **触发环境**：仅在 Windows 部署环境/客户端运行时触发，在 macOS/Linux 等标准 UTF-8 编码系统下表现正常。

### 现象 B：`errcode: 45003, errmsg: title size out of limit`
*   **现象**：文章保存为草稿失败，提示标题超限。
*   **异常点**：标题实际中文字符数非常短（如：`openclaw各大厂商现在发展的如何了`，仅 20 个字符），但被微信判定为突破了 `64` 字节上限。

---

## 2. 根源探析（Why?）

### 根源 A：Windows 隐式 Locale 转换引起的数据损坏
在原有设计中，发送草稿请求的代码为：
```python
response = requests.post(
    url, 
    params=params,
    data=json.dumps(draft_data, ensure_ascii=False), # 返回 Unicode 字符串
    headers=headers,
    timeout=self.timeout
)
```
*   **机理**：`json.dumps(..., ensure_ascii=False)` 生成一个 Python 的 Unicode 字符串。
*   **Bug 产生点**：由于直接将 Python `str` 传给 `requests` 的 `data` 参数，在 **Windows 系统**下，底层的网络数据处理可能受系统默认代码页（如 `GBK` / `CP936`）的影响进行隐式转码，导致非 ASCII 字符（如中文）发生字节截断或损坏。微信在接收并解析该被损坏的 JSON 时，无法提取出正确的 `thumb_media_id`，因此误报了 `40007`（封面图 ID 不存在/无效）。

### 根源 B：默认 JSON 序列化转义导致的体积爆表
微信公众号图文草稿 API 规定：**标题最大长度为 64 字节（UTF-8 编码）**。
*   **机理**：若直接使用 `requests.post(..., json=payload)` 发送，`requests` 内部默认使用 `json.dumps()`（即 `ensure_ascii=True`）将对象序列化为 ASCII 字符串。所有非 ASCII 的中文字符将被转义为 `\uXXXX` 格式。
*   **Bug 产生点**：
    一个中文字符在标准的 UTF-8 编码下占用 **3 字节**，但转义为 `\uXXXX` 格式会占用 **6 字符（6 字节）**。
    *   例如标题：“openclaw各大厂商现在发展的如何了”（8 个字母 + 12 个汉字）：
        *   标准 UTF-8 字节长度：`8 + 12 * 3 = 44 字节`（小于 64，安全）。
        *   `ensure_ascii=True` 格式长度：`8 + 12 * 6 = 80 字节`（大于 64，直接触发微信 API 限制报错 `45003`）。

## 3. 彻底整改方案与代码集成

我们在 [services/draft_service.py](file:///services/draft_service.py) 中应用了以下整改方案：

### 方案 1：强制手动 UTF-8 字节流序列化（解决 40007）
将请求载荷在 Python 层显式进行 UTF-8 编码为 `bytes` 后发送。Python 和 `requests` 库对于 `bytes` 类型数据将直接透传，不会进行任何 Locale 关联的二次转码：
```python
# 将 JSON 序列化并显式编码为 bytes
payload_bytes = json.dumps(draft_data, ensure_ascii=False).encode('utf-8')

response = requests.post(
    url, 
    params=params,
    data=payload_bytes,  # 传入 raw bytes 绕过一切 locale 隐式转换
    headers=headers,
    timeout=self.timeout
)
```

### 方案 2：高精度标题自动截断保护（解决 45003）
微信官方限制的是 **64字节**，而非字符数。在 `build_draft_data` 中，我们在字节维度对标题进行安全截断：
```python
# 限制标题 UTF-8 编码后的最大大小为 64 字节
encoded_title = title.encode('utf-8')
if len(encoded_title) > 64:
    logger.warning(f"标题超过64字节，将被截断。原标题: {title}")
    # 安全截取 64 字节并还原，errors='ignore' 用于丢弃末尾可能被切碎的多字节序列，防乱码
    title = encoded_title[:64].decode('utf-8', errors='ignore')
    logger.warning(f"截断后标题: {title}")
```

### 方案 3：校验规则同步修改
将 `validate_draft_data` 中原有的字符数校验改为符合微信真实的字节数校验：
```diff
- # 检查标题长度
- if len(article['title']) > 64:
+ # 检查标题长度（官方限制64字节）
+ if len(article['title'].encode('utf-8')) > 64:
      logger.error(f"文章{i+1}标题过长，超过64字节限制")
      return False
```

## ⚠️ 开发者黄金准则（如何预防同类问题）

> [!IMPORTANT]
> **在后续迭代和扩展微信 API（如：修改草稿、发布草稿、评论管理等）时，请务必严格遵守以下准则：**

1.  **绝不使用 `json=` 发送包含中文的 payload**
    凡是发送给微信的 API，只要其中有中文，请使用 `data=json.dumps(payload, ensure_ascii=False).encode('utf-8')` 的组合方式发送。
2.  **绝不直接向 `data=` 传入未编码的 `str`**
    始终先调用 `.encode('utf-8')` 将字符串显式转为 `bytes`，这在 Windows 打包成 `.exe` 运行时是防范数据乱码/格式损坏的底线手段。
3.  **处理中文字符截断时必须在 bytes 级切片**
    由于中文字符多字节的特性，直接在 `str` 层切片难以精确控制最终传输字节。请务必像方案 2 一样：先 `.encode('utf-8')`，在 `bytes` 级做切片，再通过 `.decode('utf-8', errors='ignore')` 还原，这是防止汉字截断乱码的最优解。
